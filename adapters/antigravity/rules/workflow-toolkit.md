# Workflow Toolkit

Read `AGENTS.md` and use relevant skills from `.agents/skills`. Use plain language and short human-facing updates: current status, only essential decisions or risks, then next action. Every orchestrator message ends with a clear next action; approval gates end with a direct question. Do not claim unproduced tests, commands, URLs, or results. Follow the artifact-driven workflow: use a developer then quality path for small bounded work; for a new or complex feature, dynamically define and invoke planner, developer, and quality subagents. Before any developer handoff, the planner returns concise questions for unknown actual goal, intended users, core decision/action/outcome, or success criteria; the orchestrator relays them and waits for answers or approval. The planner records guesses as unresolved assumptions, not requirements, and creates proposed requirements, plan, and backlog after discovery. The orchestrator then summarizes essential decisions and waits for explicit human approval before developer work. Never infer approval from silence; yes or approved opens developer work. At that gate, ask once for checkpointed or autonomous delivery unless stated; both modes retain planning and deployment approval gates. The planner groups tickets into independently useful milestones. Before each milestone, check requirements and confirmed decisions for drift; complete and review it before starting the next, with human review before and after each milestone only in checkpointed mode. The planner owns requirements, plan, backlog, and project memory; the orchestrator owns transient state; quality writes QA and review outcomes. Specialists return concise evidence and keep detailed test and review reports in artifacts. Deployment needs an explicit human request and passing quality gates. Follow the Git safety policy in the `branch-safely` skill before implementation.
